# 人生足迹项目技术文档

## 1. 项目概述

本项目最初是一个单页静态网页实验，历经多轮需求迭代，最终演进为一个可登录、可跨设备同步、可记录轨迹、可编辑地点名的“人生足迹”Web 应用。

当前核心能力：

1. 用户认证（注册 / 登录 / 退出，Supabase Auth）
2. 地理定位记录（浏览器 Geolocation）
3. 地图展示与轨迹模式（Leaflet + OpenStreetMap）
4. 云端数据持久化（Supabase Postgres）
5. 历史记录筛选与撤回
6. 地点名称自动解析与手动编辑持久化
7. 轨迹方向箭头（体现先后顺序）
8. 移动端与桌面端自适应布局

---

## 2. 技术栈与架构

### 2.1 前端

1. 技术形态：单页 `index.html`（HTML + CSS + 原生 JavaScript）
2. 地图库：Leaflet
3. 地图瓦片：OpenStreetMap
4. 反向地理编码：Nominatim Reverse API（地点名解析）
5. 本地缓存：`localStorage`（地点名缓存）

### 2.2 后端服务（BaaS）

1. Supabase Auth：用户注册、登录、会话管理
2. Supabase Postgres：存储足迹记录
3. RLS（Row Level Security）：按 `auth.uid() = user_id` 做数据隔离

### 2.3 部署

1. 代码托管：GitHub
2. 发布方式：GitHub Pages（`main` 分支推送后自动生效）
3. 线上地址：`https://coreyjhu-ops.github.io/cyberpunk-page/`

## 4. 关键需求与问题解决记录

### 4.1 从本地存储到跨设备同步

#### 问题

电脑记录的位置在手机看不到。

#### 根因

早期数据存储在浏览器 `localStorage`，仅限本设备本浏览器，无法跨端同步。

#### 解决方案

将记录迁移到 Supabase 数据库，改为云端读写。  
结果：同一账号在手机和电脑上可访问同一份记录。

---

### 4.2 数据归属与账号隔离

#### 问题

云端同步后，需要确保每位用户只看到自己的记录。

#### 解决方案

1. 表中使用 `user_id` 关联 `auth.users.id`
2. 启用 RLS
3. 通过策略限定 `select/insert/delete/update` 仅作用于本人

---

### 4.3 注册“看起来没反应”

#### 问题

用户点击注册后，感知上“没有效果”。

#### 根因

1. 反馈信息不足，用户不知道当前状态
2. 早期邮箱验证开关导致注册后无法立即登录

#### 解决方案

1. 增加清晰的注册/登录状态文案
2. 后续将注册改为独立弹窗，流程更清晰
3. 在用户关闭邮箱验证后，注册可直接进入登录流程

---

### 4.4 UI 遮挡问题

#### 问题

筛选控件覆盖在地图上，影响浏览。

#### 解决方案

布局重构为三栏：

1. 左侧：登录与筛选
2. 中间：地图
3. 右侧：历史记录

结果：筛选区不再遮挡地图。

---

### 4.5 地点信息可读性不足（仅经纬度）

#### 问题

历史中只有经纬度，用户难以识别地点。

#### 解决方案

1. 记录时通过 Nominatim 反向地理编码得到地点名
2. 在历史列表和地图弹窗显示地点名
3. 缺失时做后台补全并缓存
4. 将地点名持久化到数据库字段 `place_name`

---

### 4.6 用户手动纠正地点名

#### 需求

自动解析可能不准确，用户需要手动修改。

#### 实现

1. 每条历史记录增加“修改地点名”按钮
2. 弹出输入框提交新名称
3. 回写 Supabase `place_name` 字段并实时刷新界面

---

### 4.7 轨迹先后关系不清晰

#### 问题

轨迹模式只有连线，看不出方向。

#### 解决方案

在每段连线中点绘制方向箭头，按前后点位计算方位角并旋转图标。  
结果：用户可直观看到移动方向与先后顺序。

## 6. 数据模型与权限

### 6.1 表结构（核心字段）

表：`public.travel_points`

1. `id`：`uuid`，主键
2. `user_id`：`uuid`，关联 `auth.users.id`
3. `lat`：`double precision`
4. `lng`：`double precision`
5. `visited_at`：`timestamptz`
6. `created_at`：`timestamptz`
7. `place_name`：`text`（新增）

### 6.2 RLS 策略（必需）

1. `select`：用户只能读自己的记录
2. `insert`：用户只能写自己的记录
3. `delete`：用户只能删自己的记录
4. `update`：用户只能改自己的记录（用于编辑 `place_name`）

## 7. 前端核心模块说明

### 7.1 认证模块

1. `signInWithPassword`：登录
2. `signUp`：弹窗注册
3. `signOut`：退出
4. `onAuthStateChange`：会话变化后自动刷新用户数据

### 7.2 数据模块

1. 登录后拉取当前用户全部足迹
2. 新增记录时写入经纬度、时间、地点名
3. 撤回时删除最新记录
4. 修改地点名时更新数据库

### 7.3 地图模块

1. 点位渲染（全部地点）
2. 轨迹渲染（折线 + 箭头）
3. 地图范围自动调整（`fitBounds` / `setView`）

### 7.4 地点名模块

1. Nominatim 反查地名
2. 字段优先级提取（`name`、`address`、`display_name`）
3. `localStorage` 缓存避免重复请求
4. 地名补全后回写数据库（若列与策略可用）

## 8. 运行与发布流程

1. 本地修改 `index.html`
2. `git add` + `git commit`
3. `git push origin main`
4. GitHub Pages 自动发布
5. 等待缓存刷新（通常几十秒到几分钟）

---

## 9. 已知依赖与注意事项

1. 浏览器定位需 HTTPS + 用户授权
2. 反向地理编码依赖 Nominatim 网络可用性与限流策略
3. 若缺少 `place_name` 列或 `update` 策略，地点名持久化/修改会失败（前端已给出提示）
4. Supabase 的 Auth 与 RLS 配置必须与前端逻辑匹配

---

## 10. 本次迭代成果总结

相较于最初版本，项目已从“静态展示页/工具页”升级为“可登录、可同步、可追踪、可编辑的个人足迹系统”。  
核心进步体现在：

1. 数据层：从本地临时存储升级为云端持久化与权限隔离
2. 业务层：从单点记录升级为轨迹分析与地点语义化
3. 交互层：从基础按钮升级为完整注册登录流程、反馈体系与可编辑能力
4. 可用性：从单设备体验升级为跨设备一致体验
